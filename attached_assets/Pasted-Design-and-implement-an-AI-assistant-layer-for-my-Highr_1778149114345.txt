Design and implement an AI assistant layer for my Highrise multi-bot project.

Do not break existing command routing.
Do not change bot ownership.
Do not expose bot tokens.
Do not change player balances, mining rewards, poker state, blackjack state, shop prices, bank settings, or event rewards unless an authorized user explicitly confirms the action through the AI confirmation system.

Goal:
Let users talk naturally to the bots like personal assistants.

Examples:
- "how do I mine?" -> suggest /minehelp or /mine
- "show my balance" -> suggest or run /bal
- "send 100 coins to Marion" -> prepare /send Marion 100 and ask for confirmation
- "set max send to 50000" -> prepare /setmaxsend 50000 and ask for confirmation if user is admin
- "start poker" -> suggest /poker or /pokerhelp
- "what commands do you have?" -> show /help or /mycommands
- unknown requests should say no matching command exists

Create or update:
1. modules/ai_assistant.py
2. database.py if pending confirmation storage is needed
3. main.py only to add the AI assistant entry point
4. command registry/audit files only if needed for lookup

Core features:

1. Natural language command matching

Create an intent matcher that maps user text to existing commands.

It should use the existing command registry, command owner map, and known command list.

Examples:
- "show my balance" -> /bal
- "check my coins" -> /bal
- "how do I mine" -> /minehelp
- "mine for me" -> /mine
- "show my ores" -> /ores
- "send 100 to Claire" -> /send Claire 100
- "open shop" -> /shop
- "show badges" -> /shop badges
- "how to play poker" -> /pokerhelp
- "how to play blackjack" -> /bjhelp
- "events?" -> /eventhelp
- "start event double coins" -> /startevent double_coins
- "set max send to 50000" -> /setmaxsend 50000

2. Unknown command detection

If no matching command exists, reply:

"I don’t have a command for that yet. Try /help or /mycommands."

Do not hallucinate commands.
Do not invent commands.
Do not execute commands that are not registered.

3. Confirmation system

For risky commands, never execute immediately.

Create pending confirmations with:
- user_id
- username
- proposed_command
- proposed_command_args
- human_readable_action
- risk_level
- reason
- created_at
- expires_at
- status

Confirmation should expire after 60 seconds.

The bot should ask:

"Confirm: <human readable action>. Reply yes or no."

Example:
User: "send 100 coins to Claire"
Bot: "Confirm: send 100 coins to @Claire. Reply yes or no."

4. Natural yes/no confirmation

I do not want users to type /confirm yes.

When the AI assistant asks for confirmation, the user should be able to reply naturally with:

YES responses:
- yes
- y
- confirm
- do it
- proceed
- go ahead
- sure
- ok
- okay

NO responses:
- no
- n
- cancel
- stop
- nevermind
- never mind
- don’t
- do not

Rules:
1. Only treat "yes" or "no" as confirmation if the same user has a pending AI action.
2. Confirmation should only work for the user who created the pending action.
3. Confirmation should expire after 60 seconds.
4. If there is no pending action, normal messages like "yes" should be ignored or treated as normal chat.
5. Keep /confirm yes and /confirm no as backup commands, but do not require them.
6. Add /pendingaction so users can see their pending confirmation.
7. After yes, execute the proposed command through the existing command router.
8. After no, cancel the action.
9. Do not bypass permissions.
10. Do not expose bot tokens or secrets.

6. Permission checks

Before creating a confirmation, verify the user has permission for admin commands using the existing permission system.

Do not let the AI bypass permission checks.

If a normal user asks:
"set max send to 100000"

Reply:
"That is an admin bank setting. You don’t have permission to change it."

If an admin asks:
"set max send to 100000"

Reply:
"Confirm: set maximum send limit to 100,000 coins. Reply yes or no."

8. Trigger rules

The AI assistant should activate when:
- user mentions the bot by name
- user starts message with "bot,"
- user uses /ask <message>
- user uses /ai <message>
- user uses /assistant <message>

Add commands:
- /ask <message>
- /ai <message>
- /assistant <message>
- /pendingaction
- /confirm yes
- /confirm no

The /ask, /ai, and /assistant commands should be owned by host unless the message clearly belongs to a module.

9. Do not spam

Only one bot should answer an AI request.

Use existing multi_bot ownership logic.

Default AI owner should be host unless the message clearly belongs to a module.

If multiple bots could answer, prefer:
1. Mentioned bot
2. Module owner
3. Host

Do not let all bots answer the same natural language message.

10. Command existence check

Before suggesting or executing any command:
1. Check if the command exists in the registry/known commands.
2. Check if it has a route.
3. Check if it has a handler.
4. Check if the owner bot is online.
5. If owner bot is offline, respond with:
   "That module is offline right now. Try again later."

If command does not exist:
"I don’t have a command for that yet. Try /help or /mycommands."

12. Pending action behavior

Add /pendingaction.

If there is a pending action:
"Pending: <human readable action>. Reply yes or no. Expires in <seconds>s."

If none:
"You have no pending action."

If expired:
"Your pending action expired. Please ask again."

13. Execution system

When the user confirms:
- Execute the proposed command using the existing command router/handler.
- Do not duplicate command logic inside ai_assistant.py.
- Do not manually change balances/settings in the AI module.
- The AI module should prepare and confirm, then pass to the existing safe command execution path.

14. Logging

Log AI actions safely:
- username
- interpreted intent
- proposed command
- risk level
- confirmed/cancelled/expired
- timestamp

Do not log:
- bot tokens
- room ID secrets
- raw secret values
- private database credentials
