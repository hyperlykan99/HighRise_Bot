GUARANTEED NOTIFICATION FIX — Hard Cut Old DM Subscriber Pipeline

Current live bug:
Random DM messages still cause:

Alerts: ON
To unsubscribe, reply !unsub or !unsubscribe.

Screenshot shows user DMs:
Hello
.
Then bot replies Alerts: ON.

Expected:
Random DMs must get no reply and must not subscribe.

Root cause:
main.py still imports and uses old modules/subscribers.py process_incoming_dm.
subscribers.py still performs old subscriber logic before delegating to notify_system.py.
This means old subscriber/notification code can still auto-subscribe or send old replies.

Highrise SDK rule:
DMs are received through on_message(user_id, conversation_id, is_new_conversation).
DM replies use send_message(conversation_id, message, "text").
The user must DM first so the bot has conversation_id.

Final required architecture:
DM notification flow must go directly from main.py on_message to modules/notify_system.py.
Do NOT route DMs through modules/subscribers.py anymore.

PART 1 — Replace DM Entry Point
==================================================

Find main.py on_message handler.

Current likely behavior:
await process_incoming_dm(bot, user_id, conversation_id, content, is_new_conversation)

Remove/disable this for notification subscription.

New behavior:

async def on_message(self, user_id, conversation_id, is_new_conversation):
    content = extract message text from SDK event exactly as currently done
    raw = content.strip()
    lower = raw.lower()

    from modules.notify_system import process_dm_notify, is_valid_notify_dm_command

    if lower not in {"!sub", "!subscribe", "sub", "subscribe",
                     "!unsub", "!unsubscribe", "unsub", "unsubscribe"}:
        print(f"[DM HARD IGNORE] user_id={user_id[:12]} raw={raw[:60]!r}")
        return

    # Only exact sub/unsub reaches notify_system.
    await process_dm_notify(self, user_id, username, conversation_id, raw)
    return

Important:
If username is not available from on_message, use username = "" or lookup safely.
Do not block first-time DM !sub just because user is not in database.
DM !sub should still subscribe using user_id + conversation_id.
