Job-Based Subscriber Notification Bot Routing

Current issue:
Subscriber notifications are too general and are being handled by one bot.
I want notifications to come from the correct bot depending on the notification category/job.

Goal:
Route subscriber notifications to the correct bot based on category.

Example:
- Mining alerts should come from GreatestProspector.
- Fishing alerts should come from MasterAngler.
- Reward/gold/donation alerts should come from BankerBot.
- BlackJack alerts should come from BlackJackBot.
- Poker alerts should come from PokerBot.
- Events/First Hunt/updates should come from EmceeBot or EventBot.

Do not change bot tokens.
Do not change room ID.
Do not reset subscriptions.
Do not wipe notification preferences.
Do not wipe notification logs.
Do not break /notif, /notifon, /notifoff, /notifall, /subnotify, /subnotifyinvite, /subnotifystatus, or /testnotify.
Do not break mining, fishing, events, rewards, donations, blackjack, poker, or command routing.

==================================================

Add a notification category owner map.

Categories and sender bots:

mining      = GreatestProspector
fishing     = MasterAngler

==================================================
PART 2 — /subnotify ROUTING
==================================================

When Manager+ uses:

/subnotify <category> <message>

The system should:
1. Validate category.
2. Select sender bot using the category owner map.
3. Deliver the notification from that sender bot.
4. If selected bot is offline/unavailable, fallback to EmceeBot.
5. Log which bot actually sent it.

Examples:

/subnotify mining Ore Value Surge starts in 5 minutes!

Sender:
GreatestProspector

Ore Value Surge starts in 5 minutes!

==================================================
PART 3 — IN-ROOM WHISPER PRIORITY
==================================================

Keep the existing rule:

If subscribed player is currently in room:
- selected job bot sends whisper
- status = sent_whisper_in_room
- do not require DM conversation
- do not try DM first
- do not fail because conversation_id is missing

If player is not in room:
- selected job bot tries DM/conversation if supported
- if DM succeeds: status = sent_dm
- if no conversation: status = no_conversation
- if SDK does not support DM: status = unsupported_sdk
- if failed: status = failed

Important:
In-room whisper must always be priority.

==================================================
PART 4 — FALLBACK BOT
==================================================

If selected job bot is offline, unavailable, missing, or cannot send:

Fallback sender:
EmceeBot

Example:

EventBot unavailable.

Response/log:
Sender selected: EventBot
Fallback used: EmceeBot
Status: sent_whisper_in_room

Do not mark as sent unless fallback actually sends.

PART 6 — /subnotifyinvite ROUTING
==================================================

/subnotifyinvite <category> <message>

Should use the same category owner map.

If invite-style DM is supported:
- selected job bot sends invite-style message.

If invite-style DM is not supported:
- selected job bot sends normal notification.
- log invite_unsupported or normal_fallback.

If player is currently in room:
- whisper normally.
- no invite needed.

PART 10 — ROUTING / BOT LOOKUP HELPER
==================================================

Add helper:

get_notification_sender_bot(category)

Returns:
- bot instance or bot key for selected category
- fallback bot if selected is unavailable
- sender_bot_name
- original_sender_bot_name
- fallback_used true/false

It should never crash.

If selected bot cannot be found:
- fallback to EmceeBot
- log fallback_used = 1

If EmceeBot also unavailable:
- return error safely
- status = failed
- do not mark as sent
