A4 — ORE WEIGHT SYSTEM
==================================================

Add ore weights.

Every mined ore should generate a weight.

Weight should scale/multiply ore value.

Do not break existing mining rewards.
Use existing ore base value and existing mining/event boosts, then apply weight multiplier safely.

Suggested formula:
final_value = base_ore_value × weight_multiplier × existing_event_multiplier

Weight multiplier can be based on weight.

Recommended simple formula:
weight_multiplier = weight

Example:
Gold Ore base value = 500c
Weight = 3.42kg
Final value = 500 × 3.42 = 1,710c

If this is too strong for existing economy, add configurable:
weight_value_multiplier_scale

Alternative safe formula:
final_value = base_ore_value × (1 + ((weight - 1) × weight_value_multiplier_scale))

Default:
weight_value_multiplier_scale = 1.0

Default rarity weight ranges:
Common: 0.50kg to 2.00kg
Rare: 0.75kg to 3.00kg
Epic: 1.00kg to 4.00kg
Legendary: 1.50kg to 6.00kg
Mythic: 2.00kg to 8.00kg
Prismatic: 2.50kg to 10.00kg
Exotic: 3.00kg to 12.00kg

If an ore already has its own min_weight/max_weight, use that first.
Otherwise use rarity default.

Do not accidentally multiply old inventory retroactively unless explicitly intended.

PROJECT E — GOLD TIPPING UPDATE
==================================================

Goal:
BankingBot should handle all gold tipping, not EmceeBot.

Rules:
1. If any bot receives a gold tip, that receiving bot acknowledges the tip.
2. BankingBot handles the conversion, accounting, crediting, logs, limits, and final confirmation.
3. EmceeBot should not process gold tips anymore.
4. Any tipped bot can receive the gold tip, but BankingBot processes it.
5. Prevent duplicate gold tip conversion.
6. Do not change normal coin send/banking unless necessary.
7. Do not reset balances.

Flow example:

Player tips GreatestProspector gold.

GreatestProspector says:
Thank you for the gold tip! BankingBot will convert it for you.

BankingBot says:
Gold tip received from @player via GreatestProspector.
Converted 10 gold → 10,000 coins.

Gold conversion settings:
- gold_tip_enabled = 1
- coins_per_gold configurable
- default coins_per_gold should use existing rate if one exists
- if no existing rate, set safe default and report it

Commands:
- /goldtipsettings
- /setgoldrate <coins_per_gold>
- /goldtiplogs
- /mygoldtips
- /goldtipstatus

Owner:
BankingBot or Banker.

Command behavior:

/goldtipsettings:
Shows enabled status, coins_per_gold, duplicate protection status.

/setgoldrate <coins_per_gold>:
Admin/owner only.
Sets conversion rate.

/goldtiplogs:
Staff/admin only.
Shows recent gold tips.

/mygoldtips:
Shows player’s gold tip conversion history.

/goldtipstatus:
Shows whether gold tipping is enabled and current conversion rate.

Database:
gold_tip_events:
- id INTEGER PRIMARY KEY AUTOINCREMENT
- event_id TEXT UNIQUE if available
- from_user_id TEXT
- from_username TEXT
- receiving_bot TEXT
- gold_amount REAL
- coins_converted INTEGER
- conversion_rate REAL
- processed_by TEXT DEFAULT 'BankingBot'
- status TEXT
- created_at TEXT

Duplicate protection:
- Use event_id if SDK provides one.
- If no event_id, create a dedupe key:
  from_user_id + receiving_bot + gold_amount + timestamp bucket
- Do not process same tip twice.

Receiving bot acknowledgment:
Each bot that receives a tip should say a short thank you.
But actual coin conversion must be delegated to BankingBot.

BankingBot processing:
- validate tip
- convert gold to coins
- credit player balance using existing bank function
- log event
- whisper or announce confirmation

If SDK does not expose gold tip events:
- Add safe stub/log message and report:
“Gold tip event hook is not available in this SDK; handlers are ready but need event wiring.”

Do not expose secrets.
