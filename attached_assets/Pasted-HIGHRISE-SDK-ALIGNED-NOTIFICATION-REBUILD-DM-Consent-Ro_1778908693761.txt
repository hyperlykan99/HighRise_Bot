HIGHRISE SDK-ALIGNED NOTIFICATION REBUILD — DM Consent + Room Settings Only

Current issue:
The notify/subscribe system has too many bugs and old conflicting modules.

New final design based on Highrise SDK behavior:
- Highrise bots can only DM users through an existing conversation_id.
- A conversation_id exists only after the user messages the bot.
- Therefore, the first/main subscription flow must start in DM.
- Room chat can manage settings only after the user has an existing notification record.
- Random DMs must be ignored.
- Old notify/subscribe modules must stop fighting each other.

Official behavior:
DM = consent ON/OFF only.
Room = notification settings management.
Broadcasts = send only to subscribed users with category ON.

Do NOT wipe unrelated systems:
- player balances
- Luxe Tickets
- badges
- titles
- poker
- blackjack
- jail
- mining/fishing stats
- casino stats
- bank history
- user accounts

Only reset/replace notification data and notification modules.

==================================================
PART 1 — Highrise SDK Rules To Follow
==================================================

Highrise SDK DM behavior:
- Bot receives DMs through on_message(user_id, conversation_id, is_new_conversation).
- Bot sends DM/inbox messages using send_message(conversation_id, message, type, room_id).
- send_message only works with an existing conversation_id.
- Therefore, the user must DM the bot first before the bot can reliably send future inbox alerts.

Required design:
- First-time subscribe must happen by DM.
- Room !sub cannot fully subscribe a new user unless conversation_id already exists.
- Save conversation_id when user DMs !sub or !subscribe.
- Use conversation_id for future alert delivery.
- If conversation_id is missing, do not pretend inbox delivery is available.

PART 15 — SDK-Safe DM Sending
==================================================

Implement helper:

async def send_notify_dm(bot, user_id, message):
    row = get_notification_user(user_id)
    if not row or not row["conversation_id"]:
        return "no_conversation_id"
    if not row["subscribed"]:
        return "unsubscribed"
    try:
        await bot.highrise.send_message(row["conversation_id"], message[:249], "text")
        return "sent"
    except Exception as exc:
        log failure
        return "failed"

Do not use send_whisper for inbox delivery.
Use send_whisper only for room command replies.

PART 16 — DM Handler
==================================================

Use on_message DM handler.

When bot receives DM:
- identify user_id
- save conversation_id only if valid subscription command OR if existing user is already known
- if message is !sub/!subscribe/sub/subscribe:
  subscribe and reply Alerts ON
- if message is !unsub/!unsubscribe/unsub/unsubscribe:
  unsubscribe and reply Alerts OFF
- else:
  ignore silently

Host bot only should process DM notification commands.
